Why a massive dataset matters

With 32 hand-crafted roles the TF-IDF vocabulary has ~3,000 unique terms. With 124,000 real postings it expands to 50,000+ terms — IDF weights become statistically meaningful, and the LSA/BERT embedding space learns nuanced distinctions between roles like "MLOps Platform Engineer", "Quantitative Researcher", or "DeFi Protocol Developer" that simply don't exist in a curated list. A resume mentioning "Databricks" and "Delta Lake" will correctly surface "Data Lakehouse Engineer" roles it would never find in our built-in data.

🔁 The Full NLP Pipeline

1
NLTK Preprocessing
word_tokenize splits text → stopword removal (150+ common words discarded) → WordNetLemmatizer ("running"→"run", "engineers"→"engineer") → PorterStemmer ("developing"→"develop"). Tech-term normalisation runs first: C++→cplusplus, Node.js→nodejs, CI/CD→cicd.
↓
2
sklearn TF-IDF (sparse cosine)
TfidfVectorizer with word n-grams (1-2) + char n-grams (3-5) stacked into a 25k-feature sparse matrix. sublinear_tf dampens frequency extremes. Cosine similarity between resume and all job vectors is computed in a single matrix multiply.
↓
3
Sentence Embeddings (BERT or LSA)
Tier 1: sentence-transformers/all-MiniLM-L6-v2 maps text to a 384-d dense vector encoding deep semantic meaning. Tier 2 fallback: TruncatedSVD reduces the combined TF-IDF matrix to 128 dimensions (LSA), approximating latent semantic space without a GPU.
↓
4
Weighted Skill Pattern Matching (100+ skills)
Regex patterns match named skills in both resume and job. Premium skills (Python, PyTorch, Kubernetes, LLM…) score 2×. This anchors semantic scores to real employer requirements.
↓
5
Score Fusion → Final Rank
Three scores blended with tier-adaptive weights (below). Each job ranked by combined score, scaled 0–99%.

🤖 What is BERT?

BERT (Bidirectional Encoder Representations from Transformers — Google, 2018) is a deep neural network pre-trained on 3.3 billion words using two self-supervised tasks:

Task 1 — Masked Language Modelling (MLM)

15% of tokens are randomly masked. BERT predicts them using context from both sides simultaneously — making it truly bidirectional (unlike GPT which reads left-to-right only).

Softwareengineerwith5yearsof[MASK]← predict "Python"
BERT reads both directions to fill the mask — unlike earlier left-to-right language models
Task 2 — Next Sentence Prediction (NSP)

Given two sentences A and B, BERT predicts whether B actually follows A in the source document. This trains the model to understand inter-sentence relationships — critical for comparing a resume paragraph against a job description paragraph.

After pre-training, BERT's weights encode rich world knowledge. The word "Python" in "Python developer" gets a different vector than "Python snake" — context-sensitive representation that bag-of-words models can never achieve.

📐 From BERT to Sentence Embeddings

Raw BERT outputs one vector per token, not per sentence. Sentence-BERT (SBERT, 2019) adds a mean-pooling head and fine-tunes with a Siamese network architecture to produce a single fixed-size vector per sentence. The model we use:

Contrastive Training — how it learns similarity

Fine-tuned on 1 billion sentence pairs from Q&A datasets, Reddit, Wikipedia, NLI corpora. The loss function pushes semantically similar sentences together and dissimilar ones apart:

# Simplified InfoNCE / NT-Xent contrastive loss
for batch in dataloader():          # 1B sentence pairs total
    anchor, positive = batch       # e.g. ("Python dev", "ML engineer Python")
    negatives = other_sentences()  # random samples from same batch

    e_a = bert_encode(anchor)      # shape: (batch, 384)
    e_p = bert_encode(positive)    # shape: (batch, 384)
    e_n = bert_encode(negatives)   # shape: (batch, 384)

    sim_matrix = cosine_sim(e_a, cat([e_p, e_n]))

    # Pull anchor toward positive, push away from negatives
    loss = cross_entropy(sim_matrix, labels=arange(batch_size))
    loss.backward()   # AdamW, lr=2e-5, 100k steps on TPU v3-8

Result: in the 384-d embedding space, "5 years Python backend" sits close to "Senior Software Engineer Python" even with zero shared tokens.

⚡ Why Hybrid Beats Either Alone

Vocabulary mismatch — BERT's advantage

TF-IDF is bag-of-words: it scores keyword overlap. If your resume says "developed scalable microservices" and the job says "build distributed backend systems", TF-IDF scores this near-zero — no shared keywords. BERT understands they mean the same thing:

"developed scalable microservices"↔"build distributed backend systems"BERT sim = 0.87 ✓
"Python ML engineer"↔"machine learning developer Python"BERT sim = 0.92 ✓
"Python ML engineer"↔"accountant Excel finance"BERT sim = 0.11 ✓
Why we still need TF-IDF + skill matching

BERT can assign high similarity to topically different texts that share sentence structure. TF-IDF is exact — if both resume and job mention "Kubernetes", TF-IDF catches it even when BERT's attention diffuses that signal. Skill overlap ensures employer-specified requirements always have direct weight. The three signals are complementary, not redundant.
